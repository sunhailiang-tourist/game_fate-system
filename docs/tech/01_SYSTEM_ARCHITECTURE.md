# 因果修真世界 - 系统架构设计

## 一、架构概述

### 1.1 设计目标

**MVP** 优先验证「活一世 → 死 → 选胎 → 再活一世」核心循环：

- **Tick + 状态机** 世界模拟（见 `WORLD_SIMULATION_ENGINE_SPEC.md`）
- **低延迟**：关键操作响应 <200ms
- **可维护性**：模块化单体或轻量服务

**Phase 2**：百万级并发、GPT 全 NPC、完整微服务拆分。

### 1.2 架构风格（MVP）

采用 **模块化单体 + 事件驱动**，世界模拟以 Tick 引擎为核心：

```
前端层 → API网关 → 业务服务层 → 数据层 → 基础设施层
```

## 二、架构层次

### 2.1 架构分层

| 层级 | 名称 | 职责 | 技术选型 |
|------|------|------|----------|
| **L1** | 前端层 | 用户界面展示 | React/Next.js |
| **L2** | API网关 | 请求路由、认证 | Kong/APISIX |
| **L3** | 业务服务层 | 核心业务逻辑 | Go/Node.js |
| **L4** | 数据层 | 数据存储与处理 | PostgreSQL/Redis |
| **L5** | 基础设施层 | 容器与编排 | Docker/Kubernetes |

### 2.2 模块划分

| 模块 | 职责 | 服务数量 |
|------|------|----------|
| **玩家服务** | 玩家数据管理 | 3 |
| **世界服务** | 世界状态管理 | 5 |
| **战斗服务** | 战斗计算 | 10 |
| **交易服务** | 经济交易 | 3 |
| **AI服务** | AI NPC行为 | 5 |
| **轮回服务** | 选胎、灵魂档案、单命校验 | 2 |

## 四、数据流

### 4.1 数据流向

```
用户操作 → API网关 → 业务服务 → 数据存储 → 返回结果
```

### 4.2 核心数据流

| 数据流 | 路径 | 说明 |
|----------|------|------|
| **玩家登录** | 前端→网关→PlayerService→Redis→返回 | JWT认证 |
| **战斗流程** | 前端→网关→CombatService→WorldService→返回 | 实时战斗 |
| **交易流程** | 前端→网关→TradeService→PostgreSQL→返回 | 事务处理 |
| **AI行为** | AIService→WorldService→Kafka→前端 | 异步更新 |

## 五、关键技术决策

### 5.1 数据库选择

| 数据库 | 用途 | 理由 |
|--------|------|------|
| **PostgreSQL** | 核心业务数据 | 事务支持、扩展性好 |
| **Redis** | 缓存、会话 | 高性能、低延迟 |
| **MongoDB** | 非结构化数据 | 灵活的数据模型 |
| **InfluxDB** | 时间序列数据 | 实时监控 |

### 5.2 消息队列

| 队列 | 用途 | 理由 |
|------|------|------|
| **Kafka** | 实时事件流 | 高吞吐量、持久化 |
| **RabbitMQ** | 任务队列 | 可靠投递、灵活路由 |

### 5.3 缓存策略

| 缓存类型 | 策略 | TTL |
|----------|------|-----|
| **玩家数据** | 读写分离 | 5分钟 |
| **世界状态** | 实时更新 | 10秒 |
| **配置数据** | 静态缓存 | 1小时 |

## 六、安全性

### 6.1 认证授权

| 机制 | 说明 | 实现方式 |
|------|------|----------|

| 措施 | 说明 | 实现方式 |
|------|------|----------|
| **数据加密** | 敏感数据加密 | AES-256 |
| **传输加密** | HTTPS/TLS | TLS 1.3 |
| **访问控制** | 网络隔离 | 防火墙/VPC |

## 八、监控与运维

### 8.1 监控体系

| 监控类型 | 工具 | 指标 |

| 策略 | 说明 | 实现方式 |
|------|------|----------|
| **自动扩缩容** | 根据负载自动调整 | HPA |
